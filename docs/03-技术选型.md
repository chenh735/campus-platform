# 校园课程学习与项目协作平台 - 技术选型

## 1. 总体技术栈

| 层级 | 技术 | 说明 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript | 开发效率高，适合单页应用 |
| UI 组件库 | Element Plus | 表格、表单、弹窗、分页等后台和用户端都能复用 |
| 状态管理 | Pinia | 管理用户信息、Token、全局状态 |
| 路由 | Vue Router | 管理用户端和管理员端页面 |
| 后端 | Rust + Axum | 性能好，类型安全，适合构建 REST API |
| 异步运行时 | Tokio | Rust 异步生态基础 |
| 数据库 | MySQL 8 | 成熟稳定，适合课程项目和服务器部署 |
| 数据库访问 | SQLx | 编译期 SQL 检查能力强，适合 Rust + MySQL |
| 缓存与临时数据 | Redis | 保存邮箱验证码、发送频率限制和短期缓存 |
| 鉴权 | JWT | 前后端分离常见方案 |
| 文件存储 | 本地磁盘目录 | MVP 实现简单，后续可迁移 MinIO |
| 部署 | Nginx + Rust 服务 + MySQL | 简单可靠，适合云服务器部署 |

## 2. 前端选型

### 2.1 Vue 3

选择原因：

- 学习成本相对低。
- 组件化开发方便。
- 与 Vite、Element Plus、Pinia 配合成熟。
- 适合开发课程列表、资料列表、后台表格等页面。

### 2.2 Vite

选择原因：

- 启动速度快。
- 配置简单。
- 适合 Vue 3 项目。

### 2.3 TypeScript

选择原因：

- 可以为接口响应、表单数据、页面状态定义类型。
- 减少字段拼写错误。
- 后续项目规模变大时更容易维护。

### 2.4 Element Plus

选择原因：

- 提供表格、表单、分页、弹窗、菜单、上传组件。
- 用户端和管理员端都可以使用。
- 风格简洁，符合“不需要过度追求界面美观”的要求。

### 2.5 Pinia

主要用途：

- 保存登录用户信息。
- 保存 JWT Token。
- 保存角色信息。
- 控制管理员菜单显示。

## 3. 后端选型

### 3.1 Rust

选择原因：

- 类型安全，适合后端接口和数据库模型开发。
- 性能好，部署后资源占用低。
- 对文件上传、接口服务、权限控制都有成熟生态。

### 3.2 Axum

选择原因：

- 基于 Tokio 和 Tower，生态成熟。
- 路由、中间件、状态共享写法清晰。
- 适合构建 REST API。

### 3.3 SQLx

选择原因：

- 支持 MySQL。
- 可以把查询结果映射到 Rust 结构体。
- 支持连接池。
- 后续可以使用 migration 管理数据库结构。

### 3.4 JWT 鉴权

选择原因：

- 前后端分离项目常用。
- 后端不用维护大量 Session 状态。
- 可以在 Token 中保存用户 ID 和角色。

建议：

- Access Token 有效期设置为 7 天。
- 管理员接口必须每次从数据库确认用户角色和状态，避免角色变更后 Token 长期有效造成权限问题。

## 4. 数据库选型

选择 MySQL 8。

原因：

- 部署简单。
- 学习资料多。
- 支持事务、索引、外键。
- 适合用户、课程、评价、资料、项目等关系型数据。

数据库设计重点：

- 用户与课程评价是一对多。
- 课程与学习资料是一对多。
- 用户与点赞记录是一对多。
- 用户与项目招募是一对多。
- 点赞表需要唯一索引，防止重复点赞。

## 5. 文件存储选型

### 5.1 MVP 阶段：服务器本地磁盘

文件保存路径示例：

```text
/var/www/campus-platform/uploads/materials/
```

数据库只保存：

- 文件原始名称
- 文件存储路径
- 文件类型
- 文件大小
- 上传用户
- 所属课程

优点：

- 实现简单。
- 不需要额外部署对象存储。
- 适合初版项目。

### 5.2 后续扩展：MinIO

当资料数量增多时，可以迁移到 MinIO。

迁移方向：

- 文件上传接口保持不变。
- 后端把本地存储实现替换为对象存储实现。
- 数据库中的 file_path 改为 object_key 或 URL。

## 6. Redis 与邮箱验证码

项目需要使用 Redis，主要承担临时数据和轻量缓存能力。

Redis 使用场景：

- 保存邮箱验证码，设置 10 分钟过期时间。
- 限制同一邮箱发送验证码频率，例如 60 秒一次。
- 限制同一 IP 短时间内发送验证码次数，减少恶意请求。
- 可选缓存课程列表、课程评分统计等读取频率较高的数据。

后端 Rust 可使用 `redis` 或 `deadpool-redis` 连接 Redis。MVP 推荐 `deadpool-redis`，方便在 Axum 中通过连接池复用 Redis 连接。

后端需要配置：

```env
REDIS_URL=redis://127.0.0.1:6379
```

验证码 Key 设计示例：

```text
email_code:register:{email}       -> 验证码，TTL 10 分钟
email_code_cooldown:{email}       -> 发送冷却标记，TTL 60 秒
email_code_ip_limit:{ip}:{date}   -> IP 当日发送次数，TTL 1 天
```

## 7. 邮箱验证码

建议使用 SMTP 发送验证码。

后端需要配置：

```env
SMTP_HOST=smtp.example.com
SMTP_PORT=465
SMTP_USERNAME=example@example.com
SMTP_PASSWORD=your_password
SMTP_FROM=example@example.com
```

说明：

- 实际部署时需要准备一个可发送邮件的邮箱。
- 验证码不再保存到 MySQL，统一保存到 Redis。
- 验证码有效期建议 10 分钟，由 Redis TTL 控制。
- 同一邮箱发送验证码需要限制频率，例如 60 秒一次，由 Redis 冷却 Key 控制。

## 9. 部署选型

### 8.1 推荐部署方式

服务器上运行：

- Nginx：托管前端静态文件，反向代理 `/api` 到后端。
- Rust 后端：运行在 `127.0.0.1:8080`。
- MySQL：运行在服务器或独立数据库服务。
- Redis：运行在服务器或独立 Redis 服务，用于验证码、限流和缓存。
- uploads 目录：保存学习资料文件。

### 8.2 后端运行方式

可选方式：

1. systemd 运行 Rust 二进制文件，MySQL 和 Redis 使用独立服务
2. Docker Compose 运行后端、MySQL 和 Redis

MVP 推荐 systemd，部署简单。

### 8.3 Nginx 作用

- 访问 `/` 时返回前端页面。
- 访问 `/api` 时转发到 Rust 后端。
- 访问 `/uploads` 时提供静态文件下载，或转发给后端鉴权后下载。

为了控制权限，学习资料下载建议由后端接口处理，不直接暴露整个 uploads 目录。

## 10. 为什么不选择更复杂方案

暂不引入以下技术：

- 微服务：项目规模不大，单体后端更合适。
- Elasticsearch：课程和项目搜索可以先使用 MySQL LIKE。
- 即时聊天：会显著增加复杂度，初版使用申请记录和联系方式即可。
- 对象存储：初版本地磁盘足够，后续可迁移。
